🧩 What Went Wrong

When you resize a pane, the XAML layout engine fires SizeChanged events. Each one tells the terminal core "hey, you're a different shape now!" The core dutifully calls TextBuffer::Reflow to rewrap all the text for the new dimensions.

The problem? Our monkey was rapidly resizing panes with Alt+Shift+Arrow while TerminalStress was flooding the buffer with emoji. Multiple SizeChanged events piled up faster than Reflow could finish. The buffer dimensions and row storage fell out of sync, and _getRow() tried to access a row index that overflowed a 32-bit integer. The gsl::narrow_cast safety net caught it and triggered a fail-fast breakpoint. 💣

🧩 What Went Wrong

When panes are squeezed to near-zero dimensions in a small window, the XAML layout manager enters a feedback loop. Each resize triggers a SizeChanged event, which recalculates pane sizes, which triggers more SizeChanged events... The UI thread gets trapped in an infinite recalculation cycle and never gets a chance to process user input. 🔄♾️

The monkey already has a mitigation for this (max_pane_resize_repeats=5) to avoid the known issue #19996, but certain window sizes still slip through the cracks.

🧩 How It Differs From Specimen #2

Specimen #2 is a geometric problem — small window + tiny panes = layout feedback loop. Specimen #3 is a throughput problem — enormous buffers + reflow = the UI thread drowns in work. They produce different WER bucket hashes, live in different parts of the code, and are triggered by completely different conditions. Two bugs, wearing different disguises. 🎭

🎯 The Common Thread

All three bugs share a villain: TextBuffer::Reflow. 🦹

When you resize a pane, the XAML layout engine fires SizeChanged events that cascade into buffer reflow operations. These operations rewrap every line of text for the new terminal dimensions. They're expensive, they run on the UI thread, and they're not robust against re-entrant or overlapping calls.

Our monkey found three ways to break this:

🪲 Specimen #1: Rapid resizes cause overlapping reflows → buffer dimensions go inconsistent → row index overflow → crash
🦗 Specimen #2: Tiny window + resize = layout feedback loop → UI thread deadlocks on itself
🐜 Specimen #3: Huge buffers + resize = reflow takes forever → UI thread starved of message pump time

📝 Field Notes for the Devs

🐛 Note 1 — File This Upstream

Specimen #1 is a real crash with a full call stack and a 598MB dump. Ship it to microsoft/terminal with reproduction seed --seed 11112222 --action-profile buffer-chaos.

🔧 Note 2 — Debounce the Reflows

The function is called _refreshSizeUnderLock, but the XAML layout pass can fire re-entrant SizeChanged events before the lock is released. A coalescing resize mechanism (debounce multiple events into one reflow) would fix or mitigate all three bugs.

🐒 Note 3 — Keep the Monkeys Running

Use --action-profile novelty-hunt to spread instances across different profiles. Mix in shell diversity (pwsh, cmd, bash) for different rendering paths. The bugs are in the resize code — keep poking at it from different angles.
