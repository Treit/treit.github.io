Root Cause Analysis

During a pane resize operation, the XAML layout engine fires a SizeChanged event. This triggers ControlCore::SizeOrScaleChanged, which calls Terminal::UserResize. The resize causes TextBuffer::Reflow to restructure the text buffer for the new dimensions.

Inside Reflow, the code calls _getRow() to access a row by index. The gsl::narrow_cast<int> inside _getRow detects that the row index overflows a 32-bit integer boundary — indicating the buffer dimensions have become inconsistent during the resize. The narrow_cast fires a fail-fast breakpoint (exception 0x80000003).

This is a race condition: multiple concurrent resize events (from rapid Alt+Shift+Arrow pane resizing combined with window resizing) cause the buffer dimensions and the row storage to fall out of sync. The Reflow operation is not fully atomic with respect to concurrent SizeChanged events from the XAML layout manager.

Root Cause Analysis

The sequence that triggers this bug is: resize the WT window to a small size, then rapidly resize panes with Alt+Shift+Arrow. When the window is small and contains multiple split panes, the resize math can result in panes being squeezed to zero or near-zero dimensions.

The UI thread enters a tight loop in the XAML layout manager, recalculating pane sizes that cascade into additional SizeChanged events. This creates a feedback loop where the UI thread is entirely consumed by layout recalculation and never returns to pump messages — causing the "Not Responding" state.

Note: This is likely related to GitHub Issue #19996 — "Terminal hangs when resizing pane to minimum width." The monkey already has a mitigation (max_pane_resize_repeats=5) but the bug can still be triggered with certain window sizes.

Root Cause Analysis

This hang differs from Bug #2 in that it occurs under extreme buffer throughput rather than small-window geometry. Multiple TerminalStress processes are flooding different panes with high-speed random Unicode output (including emoji sequences and cursor repositioning) while the monkey simultaneously resizes panes and clears buffers.

The TextBuffer::Reflow operation during resize must walk every row of every pane's buffer. With large scrollback buffers filled by TerminalStress (each pane could have 9,000+ rows of output), the reflow becomes O(rows × columns) per pane. Combined with XAML layout recalculation across multiple split panes, the UI thread is blocked for long enough to trigger the Windows "Not Responding" detection (MoAppHang).

Distinguishing factor from Bug #2: Bug #2 is a geometric feedback loop (small window + resize). Bug #3 is a throughput saturation issue (large buffers + reflow cost). They produce different WER bucket hashes, confirming they are distinct code paths.

🎯 Summary & Recommendations

3 Unique Bugs Found — All in Pane Resize / Buffer Reflow Path

All three bugs share a common theme: the TextBuffer::Reflow operation during pane resize is fragile under concurrent or extreme conditions. The XAML layout engine fires SizeChanged events that cascade into buffer reflow operations which are not robust against re-entrant or overlapping calls.

Recommendation 1 — File upstream issues

Bug #1 (TextBuffer::_getRow bounds check failure) is a definitive crash with a full call stack. This should be filed at microsoft/terminal with the crash dump and reproduction seed (--seed 11112222 --action-profile buffer-chaos).

Recommendation 2 — Investigate Reflow atomicity

The Reflow operation in textBuffer.cpp should be examined for thread safety. The _refreshSizeUnderLock name suggests locking is attempted, but the XAML layout pass may trigger re-entrant calls before the lock is released. A coalescing resize mechanism (debounce multiple SizeChanged events into one reflow) could mitigate all three bugs.

Recommendation 3 — Continue stress testing with broader profiles

Use --action-profile novelty-hunt with multiple instances to cover different code paths. The new shell-aware commands and profile-diverse tabs/panes will stress different rendering paths (emoji in PowerShell, ANSI in bash, VT sequences in cmd).
